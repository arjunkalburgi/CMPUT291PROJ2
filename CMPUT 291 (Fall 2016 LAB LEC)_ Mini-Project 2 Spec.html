Mini-Project 2 Spec

CMPUT291 - Fall 2015

Mini-Project 2


Due : November 25th at 5pm 


Clarifications:

You are responsible for monitoring the course discussion forum in eclass and this section of the project specification for more details or clarifications. No clarification will be posted after 5pm on November 24th.

  • Specification update November 7 (see the description below for the exact wording):
    • Simplified the input requirements: Only one input relation schema has to be decomposed at a time.
    • The BCNF decomposition has to be applied only to the input relation (until it is completely in BCNF), but not to the result of the 3NF decomposition.
  • November 16: updated the listing of the content of Input_R1 and Output_R1_BHCK in the description below to reflect the updated example input database.
  • November 16: Here is an example output of a BCNF decomposition and an example 3NF decomposition of the given input schema in the example database. Note, however, this is just one possible outcome. There is typically more than one possible correct output, depending on the order in which FDs are processed! The given BCNF decomposition is derived using our BCNF decomposition algorithm and testing the FDs in the same order as they are listed on slide 51. This decomposition is not dependency preserving (C=>E is not preserved). If you process them in a different order, you may end up with a different decomposition, which can still be correct.
  • November 17:  You don't need to worry about potential naming conflicts that could occur if you do a 3NF and a BCNF decomposition of the same input table, one after the other (both may lead to a table with the same name according to our naming scheme). All we expect regarding decompositions is that the user should initially be able to choose either a 3NF or a BCNF decomposition of the input schema. After the decomposition of the schema, the user should then have the option to decompose the given instance according to that decomposition. During demo time, we will only do one decomposition at a time. If we want to see the other decomposition for the same input table, we will start with a "fresh" copy of the input database. 


Introduction

The goal of this project is to deepen your understanding of database normalization and practice working with sqlite3 in Python for input and output. 80% of the project mark would be assigned to your implementation which will be assessed in a demo session. Another 15% of the mark will be assigned for the documentation and quality of your source code and for your design document. 5% of the mark is assigned for your project task break-down and your group coordination.


Group work policy

You will be doing this project with one or two other partners from the 291 class. Your group information from mini-project 1 is copied to mini-project 2 groups page on the assumption that you would be working in the same group. If you decide to change groups, please notify the instructor. It is assumed that all group members contribute somewhat equally to the project, hence they will receive the same mark. In case of difficulties within a group and when a partner is not lifting his/her weight, make sure to document all your contributions. If there is a break-up, each group member will get credit only for his/her portion of the work completed (losing the mark for any work either not completed or completed by the partner). For that reason, a break-up should be your last resort.


Tasks

You are given a database file, which will contain the information about an input schema and an instance of the schema. Your job is to implement functionality that implements a normalization and decomposition of the given schema and instance into 3NF and BCNF normal forms. An example database is  in the file MiniProject2-InputExample.db - which you can download by clicking on the link (and, which is, by the way, the example used in class for 3NF normalization -see slide 51).

You should write an application that provides a simple user interface for performing the following tasks:


  • Connecting to an input database and reading the input about a relation schema to be normalized

An input database file contains one relation schemas to be normalized. For instance, the schema of the given example database is as following:

sqlite> .schema
CREATE TABLE Input_R1(
  A INT,
  B INT,
  C INT,
  D INT,
  E INT,
  F TEXT,
  G TEXT,
  H REAL,
  K REAL
);
CREATE TABLE Input_FDS_R1(LHS TEXT,RHS TEXT);

This example contains information about only one relation schema to be normalized. In general, an input database can contain multiple relation schemas to be normalized. Each such relation schema R is represented by two tables. We will use the following naming convention: a data input relation name starts with 'Input_' followed by the name (e.g. 'Input_R'), and the corresponding table with the functional dependencies starts with 'Input_FDs_' (e.g., 'InputFDS_R). Besides those tables, the database may contain other tables that are used during the demo process. The table Input_R has all the columns of the relation and contains the data (i.e., it is a table instance); Input_FDS_R contains the the functional dependencies that hold on R, one row for each functional dependency f: LHS => RHS.  LHS and RHS are left hand side and the right hand side of f, respectively, and they store as a string, a list of attributes. The given example database has 7 FDs associated with R1:

sqlite> SELECT * FROM Input_FDS_R1;
LHS|RHS
A,B,H|C,K
A|D
C|E
B,G,H|F
F|A,D
E|F
B,H|E

These correspond to the functional dependencies ABH => CK, A => D, C => E, BGH =>F, etc. 

The "data tables" also contain rows, for instance, the given table Input_R1 contains the following rows:

Your program should connect to an input database provided by a user, and read the information about the tables' attributes and functional dependencies into appropriate data structures, so that you can perform normalization as specified. Tip: to get information about the tables currently in a database you should query the table SQLITE_MASTER (sqlite3 shell commands like '.schema' are not available from within the Python module.)


  • Normalization of the schema(s)

After connecting to an input database, your user interface should allow the user to do the following

    • Synthesize a 3NF schema for the given input table, using the algorithm presented in class and described in the textbook. 
    • Decompose the given input table into BCNF according the BCNF decomposition algorithm presented in class and described in the textbook. At the end it should be indicated if the resulting BCNF decomposition is dependency preserving. 
    • For each of the resulting new schemas, create tables similar to the input format, one for the data, and one for the the functional dependencies that hold on the new relation. The naming convention for the new schemas, which you must follow, is the following: each name of a data table has to start with 'Output_' followed by the name of the corresponding input table (without the 'Input_' prefix), followed by an underscore '_', followed by the concatenation of the attributes in the new table. For instance, if one of the tables in the decomposition of 'Input_R1' has only the attributes B, H, C, K, then the name of that table should be 'Output_R1_BHCK'. For each new data table, also include in the create table command the primary key constraint that can be derived from the way the decomposition was done. In addition, you must also create a table to store the functional dependencies of the new data table. The name of this table must start with 'Output_FDs_' followed by the corresponding data output table name (without the 'Output_' prefix). In the example where the new data table is called 'Output_R1_BHCK', the corresponding table for the functional dependencies must be called 'Output_FDS_R1_BHCK' and must have two columns named 'LHS', and 'RHS' storing the functional dependencies in the same way as in the input. 
    • After a decomposition of a schema into 3NF or BCNF, the user should have the option to decompose the original table instance according to the schema decomposition, i.e., fill the new data tables according to the decomposition, using the data in the input data table.  For instance, given the example input table Input_R1 (with the rows shown above) and a new output table Output_R1_BHCK, the content of Output_R1_BHCK should be as following:

                        • Functionality related to normalization

                        Your program should also expose the following functionality (which you will need in your implementation of the normalization) to the user:  

                        1) Computation of the attribute closure of a set of attributes X with respect to a set of functional dependencies F, specified by the user as input: To provide a set X = {A, B, C}, the user should just be able to type 'A,B,C'. In order to provide a set of functional dependencies, the user should be able to provide a list of one or more table names in the current database that contain functional dependencies (in the same format as tables with the naming scheme Input_FDS_R), and the union of these tables given by the user is then considered as F.

                        2) Determine whether two sets of functional dependencies F1 and F2 are equivalent. Both sets of functional dependencies should be user-specified. For both F1 and F2, the user should be able to provide a list of names of tables containing functional dependencies, whose union of entries should be considered as F1 and as F2, respectively. 


                        Testing

                        At demo time, your code will be tested under a TA account. You will be given the name of a database file that contains the input for the normalization, and will be asked 

                        The demo will be run using the source code submitted and nothing else. Make sure your submission includes every file that is needed. There will be a limited time for each demo. Every group will book a time slot convenient to all group members to demo their projects. At demo time, all group members must be present. The TA will be asking you to perform various tasks and show how your application is handling each task. A mark will be assigned to your demo on the spot after the testing.


                        Instructions for Submissions

                        Your submission includes (1) the Python source code for your mini-project and and (2) the design document. The files are submitted as follows:

                        • Create a single gzipped tar file with your source code and documentation. Name the file prj2code.tgz.
                        • Submit your project tarfile in the project submission site by the due date given at the top of this page.

                        Your report must be type-written and also be submitted in hardcopy at the designated drop boxes located on the first floor of CSC building, across from the room 1-45. Your report cannot exceed 4 pages in length plus one additional cover page. All project members are expected to submit their code online but only one copy of the project report will be submitted.

                        The report should include (a) a general overview of your system with a small user guide, (b) a description of your algorithm for each part of the mini-project, (c) your testing strategy, and (d) your group work break-down strategy. The general overview of the system gives a high level introduction and may include a diagram showing the flow of data between different components.  The user guide should have instructions for running your code. The testing strategy discusses your general strategy for testing, with the scenarios being tested and the coverage of your test cases. The group work strategy must list the break-down of the work items among partners, both the time spent (an estimate) and the progress made by each partner, and your method of coordination to keep the project on track. The design document should also include any assumption you have made or any possible limitations your code may have.
